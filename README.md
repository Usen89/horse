# 🐎 Табун-Реестр

### Система управления коневодческим хозяйством

Профессиональный зоотехнический учёт: реестр лошадей, косяки и табуны, учёт потомства, вакцинация, откорм и архив забоя.

## О проекте

**Табун-Реестр** — веб-приложение для управления табунным коневодческим хозяйством (на примере хозяйства «Алтын Тулпар»). Приложение помогает зоотехникам и владельцам вести полный учёт поголовья с учётом традиционной казахской возрастной классификации лошадей (Құлын, Жабағы, Тай, Байтал, Құнан, Дөнен, Бие, Айғыр).

Все данные хранятся **локально в браузере** (localStorage) — приложение работает офлайн, без сервера и без передачи данных наружу.

## Возможности

| Модуль | Назначение |
| --- | --- |
| 📊 **Главная панель** | Сводная статистика по хозяйству: поголовье, приплод, вакцинация, откорм |
| 🗂️ **Реестр лошадей** | Карточки животных: масть, дата рождения, пол, владелец, статус, фото |
| 👥 **Косяки и табуны** | Группировка лошадей по косякам во главе с жеребцом-производителем |
| 🍽️ **Группа откорма** | Учёт интенсивного откорма, контроль привесов |
| 👶 **Маточное поголовье** | Кобылы, беременность, ожидаемая и фактическая выжеребка, приплод |
| 🛡️ **Вакцинация** | График прививок, сроки ревакцинации, контроль просрочек |
| 💀 **Архив забоя (согым)** | Учёт забоя: вес, выход мяса, выручка |
| 🌳 **Потомство** | Интерактивная карта потомков лошади: дети, внуки, правнуки |
| 📜 **История хозяйства** | Хронология ключевых событий |
| 👤 **Администраторы** | Несколько профилей (зоотехник, ветеринар) с переключением сессий |

## Запуск локально

**Требования:** [Node.js](https://nodejs.org/) 18+

```bash
# 1. Установить зависимости
npm install

# 2. (Опционально) указать ключ Gemini API в .env.local
#    cp .env.example .env.local  и вписать GEMINI_API_KEY

# 3. Запустить dev-сервер
npm run dev
```

Приложение откроется на `http://localhost:3000`.

### Прочие команды

```bash
npm run build    # production-сборка в dist/
npm run preview  # локальный просмотр собранной версии
npm run lint     # проверка типов (tsc --noEmit)
```

## Деплой (GitHub Pages)

Приложение автоматически публикуется на **GitHub Pages** при каждом пуше в ветку `main` —
через workflow [`.github/workflows/deploy.yml`](.github/workflows/deploy.yml).

🔗 **Живая версия:** https://usen89.github.io/horse/

Как это работает:

- CI собирает проект с базовым путём `/horse/` (через переменную `DEPLOY_BASE`), поэтому
  локальный запуск и AI Studio продолжают работать с корневым путём `/` без изменений.
- Готовая сборка `dist/` публикуется на Pages.

Первичная настройка (один раз): в репозитории **Settings → Pages → Build and deployment →
Source** выбрать **GitHub Actions**.

## Данные и приватность

Хозяйственные данные не покидают устройство — они хранятся в `localStorage` вашего браузера. Для переноса на другое устройство или создания резервной копии используйте функцию **Экспорт / Импорт данных** в приложении.

## Облачная база данных (Supabase)

Приложение поддерживает синхронизацию с собственной облачной базой на [Supabase](https://supabase.com) (бесплатный тариф) — данные доступны с любого устройства.

**Настройка (один раз, ~5 минут):**

1. Зарегистрируйтесь на [supabase.com](https://supabase.com) и создайте проект (New project, регион — любой ближайший, например Frankfurt).
2. В проекте откройте **SQL Editor → New query**, вставьте целиком содержимое файла
   [`supabase/schema.sql`](supabase/schema.sql) и нажмите **Run** — создадутся все таблицы
   (лошади, косяки, вакцинации, откорм, забой, администраторы).
3. Откройте **Settings → API** и скопируйте два значения: **Project URL** и **anon (public) key**.
4. В приложении: **Профиль → Настройки → Облачная база данных (Supabase)** — вставьте оба
   значения и нажмите **«Отправить в облако»**.

**Использование:**

- **Отправить в облако** — выгружает все локальные данные в Supabase (данные в облаке заменяются).
- **Загрузить из облака** — восстанавливает данные из Supabase на текущем устройстве
  (локальные данные заменяются).

> ⚠️ Anon key даёт полный доступ к вашей базе (политики RLS открыты для простоты
> однопользовательского сценария) — не публикуйте его. Для многопользовательского доступа
> подключите Supabase Auth и ужесточите политики (см. комментарии в `schema.sql`).
